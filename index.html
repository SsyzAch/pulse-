11.1 Circuit Breaker

The API Gateway and all inter-service gRPC clients implement circuit breakerCircuit Breaker
Like a fuse in your home — if a downstream service keeps failing, the breaker "trips" and stops calls to it, preventing failure from cascading everywhere.s. Each circuit monitors the error rate over a 10-second rolling window. The circuit opens when the error rate exceeds 50% with at least 20 requests in the window. After 30 seconds in open state, it enters half-open, allows a single probe request, and closes if successful.

11.2 Retry with Exponential Backoff

Kafka consumers implement retries: initial wait 100ms, multiplier 2.0, jitter factor 0.2, maximum wait 30 seconds, maximum attempts 5. After 5 failures the message is sent to a dead-letter topic (suffixed .dlq) with the original message, error details, and retry count appended as headers.

14.1 Algorithm
  • Each skill node is embedded into a 384-dimensional vector using sentence-transformers (all-MiniLM-L6-v2). Node embeddings are static, computed offline.
  • Each user's progress is represented as a weighted centroid of completed node embeddings, with recent completions weighted via exponential decay.
  • Gap vector = ideal_learner_state_vector − user_current_state_vector. Ideal state is derived from completion patterns of users who reached the final phase.
  • Candidate next nodes ranked by cosine similarity to the gap vector, with a multiplier for required or fintech-specific nodes.
  • Top 3 candidates cached in Redis per user for 10 minutes.

A1Flow A — Fundamentals to Backend Engineering

This is the foundation path. Every backend engineer in the world walks this sequence, in roughly this order. Nothing here is optional — each step depends on the one before it.

01Programming Language
Pick Python, Go, or Java. Learn to write correct programs, handle errors, model data, and think in functions. This is the tool you will use for every other topic on this list.
Start here. No shortcuts.
02Data Structures & Algorithms
Arrays, hash maps, queues, trees, graphs. You need these because databases use B-trees, order books use priority queues, routing uses graphs, and caches use hash maps. DSA is not academic — it is the vocabulary of systems.
03Linux & Operating Systems
Every backend service runs on Linux. Understanding processes, file systems, I/O modes, and networking tools means you can deploy, debug, and tune anything. Without Linux fluency you are dependent on others to run what you build.
04Networking (TCP/IP, HTTP, DNS)
Every API call, payment request, and market data packet travels over a network. Knowing how TCP works, what HTTP does, and how TLS protects data is the difference between a developer who guesses and one who debugs correctly.
05Relational Databases & SQL
All financial data lives in relational databases. ACID guarantees, transactions, isolation levels, and indexes are not optional in fintech — a payment that partially commits is catastrophic. PostgreSQL is the industry standard for a reason.
06APIs & Web Frameworks
REST, gRPC, and message protocols are how systems talk. Building APIs on top of a framework (FastAPI, Spring Boot, Gin) is the primary daily activity of a backend engineer. At this point you can build real, deployable services.
07Authentication & Security Basics
JWT, OAuth, rate limiting, and input validation protect what you build. No production system runs without auth. In fintech, MFA is a compliance requirement, not a nice-to-have.
08Containers & CI/CD
Docker + Kubernetes + GitHub Actions is the production deployment stack at every modern fintech company. Understanding containers means you can run the same service locally, in staging, and in production without "works on my machine" problems.
✓Backend Engineer
At this point you can build APIs, work with databases, deploy containerised services, and handle authentication. You are a credible backend engineer ready for fintech specialisation.

A4Flow D — Reliability & Scale Path

A service that works once is not production-ready. This path shows how a basic backend service becomes a distributed, observable, fault-tolerant system that can handle real load without falling over.

🗄️
1. Caching (Redis)
What it is
A fast in-memory store sitting in front of your database. Returns data in microseconds instead of milliseconds.
Connects to
Session storage → Rate limiting → Leaderboards → Pub/sub for real-time updates
Why it matters
Without caching, every user request hits the database. At 10,000 requests/second this breaks instantly.
RedisMemcachedCDN
📨
2. Message Queues (Kafka)
What it is
An event bus that decouples producers from consumers. Every node completion publishes an event; downstream services react asynchronously.
Connects to
Event sourcing → XP ledger → Streak service → Analytics → Audit service
Why it matters
Synchronous calls create tight coupling and fragility. Kafka makes the system resilient to individual service failures.
KafkaRedpandaNATS
🔄
3. Replication & Sharding
What it is
Spreading data across multiple database instances so no single server is a bottleneck or single point of failure.
Connects to
Read replicas for analytics → Hash sharding by user_id → Multi-AZ for disaster recovery
Why it matters
At 1M users, a single PostgreSQL server cannot handle all writes. Partitioning by user_id distributes load evenly.
PostgreSQL partitionsCDC

4. Circuit Breakers & Resilience
What it is
When a downstream service fails, a circuit breaker stops sending requests to it — preventing cascade failures from taking down the whole system.
Connects to
Retry with backoff → Bulkhead isolation → Load shedding → Graceful degradation
Why it matters
A payment system that takes down its entire infrastructure because one service is slow is unacceptable in production.
